Rules of the Road

  • Classifiers must output clean labels. One word, no explanation. If your classifier reasons through its answer, your router can't read it.
  • The router is dumb logic. It doesn't think. If A then B. That's its job. All the intelligence lives in the classifier.
  • Design on paper first. If you can't draw your logic branches on one page, the system is too complex to build. Simplify before you wire.
  • Test edge cases, not just happy paths. What breaks your classifier? What inputs don't fit your categories? Find those now.

Brief 1

The Linear Workflow Problem

Why One-Size-Fits-All Doesn't Work

Your Mission 3 workflow is fast. It's not smart. That's the problem we fix today.

A linear workflow treats every input the same way. Customer writes praise. System sends response A. Customer writes complaint. System sends response A. That's broken.

Think about how a smart receptionist works. A call comes in. They listen first. "Is this a new lead or an angry customer?" Once they know the answer, they pick the right script. They transfer to the right department. They change their tone. They adjust their urgency.

Your workflow needs the same intelligence. The mental model is simple:

  • First listen.
  • Then decide.
  • Then respond.

Right now you're skipping the listen and decide steps. You're jumping straight to respond.

Every input gets the same treatment. That works if every input is identical. But they're not. Praise should get a warm thank you. Complaints should get an apology and a solution. Suggestions should get documentation and a timeline. One generic response can't do all three well.

A linear workflow is blind to nuance. It's not your fault. You haven't added the decision layer yet.

Worked Example: Jordan's Customer Feedback Automation

Jordan works in customer success at a retail company. In Mission 3, Jordan built a workflow that triggers whenever customer feedback comes in. The workflow reads the feedback. Packaged AI writes a response. The response goes out. It logs the interaction.

The problem: every customer gets the same template response. "Thank you for reaching out. We appreciate your feedback and will review it." Send. Log. Done.

Without Logic Layer

Glowing review: "Your team was amazing. Best service ever."

Response: "Thank you for reaching out. We appreciate your feedback and will review it."

The praise customer feels brushed off.

With Logic Layer

Glowing review: "Your team was amazing. Best service ever."

Classified As: PRAISE

Response: "Thank you so much! Your kind words mean everything to our team. Would you consider sharing this as a review?"

Personalized. Appropriate. Effective.

A furious complaint gets the same generic response. "I've been waiting three days for my package and have no idea where it is. This is ridiculous." Response: "Thank you for reaching out. We appreciate your feedback and will review it."

Both customers got the same thing. The praise customer feels like they're being brushed off. The complaint customer feels like they're being ignored. Neither is satisfied.

Key Insight

Before AI responds, it needs to know what it's responding to. Classification comes before response. Always. This is a leadership principle, not a technical one. It's the difference between hearing and listening. A linear system hears input. A smart system listens first, understands context, then chooses a response. That's maturity in automation.

Brief 2

Classify, Route, Respond

The Three-Step Intelligence Layer

The pattern has three steps. Learn it once. You'll use it everywhere.

Step 1: Classify

The classifier is an AI step with one simple job: read the input, apply a label, output that label and nothing else.

System prompt: "Classify this feedback. Output ONLY one word: PRAISE, COMPLAINT, or SUGGESTION. No explanation. No punctuation. Just the word."

That's tight. No reasoning. No explanation. One clean label.

Why? Because your router needs to read that label and route based on it. If the classifier starts explaining itself ("This is clearly PRAISE because the customer said..."), your router can't parse that. It gets confused. The routing logic breaks.

Clean input. Clean output. One word. That's the requirement.

Step 2: Route

The router is simpler than the classifier because it's not AI. It's logic. If-then. That's all.

  • If label = PRAISE, go to path A.
  • If label = COMPLAINT, go to path B.
  • If label = SUGGESTION, go to path C.

The router is traffic control. No thinking. Just direction. Which path does this labeled input belong in? Route it there.

This happens in your automation platform. Make. Zapier. Whatever you use. You set up a conditional step. If the classifier output contains the word "PRAISE", route to the praise response step. Simple.

Step 3: Respond

Then each path has its own response step. Same structure. Different AI prompt for each path.

Praise path has an AI prompt designed for happy customers: "This customer left praise. Write a warm, genuine thank you. Keep it brief. Ask if they'd consider sharing this as a review."

Complaint path has a prompt designed for problem-solving: "This customer reported a problem. Start with a genuine apology. Ask what we can do to help. Offer a specific next step."

Suggestion path has a prompt designed for capturing ideas: "This customer shared an idea. Thank them. Confirm you've documented it. Tell them how ideas are reviewed."

Same five-part workflow from Mission 3 (Trigger, Data, AI, Action, Log). But now there's intelligence in the middle. Three classifier categories. Three router paths. Three responses, each tuned for its input type.

One workflow. Three different behaviors. That's the pattern.

Worked Example: Jordan's Complete Three-Step Build

Jordan is building this for customer feedback.

Classifier Step

System prompt: "You are a feedback classifier. Read the following feedback. Classify it as PRAISE, COMPLAINT, or SUGGESTION. Output ONLY one word. No explanation. No punctuation. Just the word."

Test inputs and outputs:

  • Input: "Your team was so helpful on my last order!" Output: PRAISE
  • Input: "My package is late again and I still don't have tracking." Output: COMPLAINT
  • Input: "Have you ever thought about offering a subscription?" Output: SUGGESTION
  • Input: "I love your product!" Output: PRAISE
  • Input: "The app keeps crashing on login." Output: COMPLAINT

Clean outputs. Consistent. One word each. The classifier is working.

Router Step

Condition Route To
Classifier output = "PRAISE" Praise Response path
Classifier output = "COMPLAINT" Complaint Response path
Classifier output = "SUGGESTION" Suggestion Response path

Jordan sets this up in Make (or Zapier). Conditional logic. Simple and clean.

Response Paths

Praise Response Path: "A customer left positive feedback. Write a warm, genuine thank you. Keep it brief. Two sentences max. End by asking if they'd consider sharing this as a public review. Sound authentic, not corporate."

Sample output: "Thank you so much! Your kind words mean everything to our team. Would you consider sharing your experience on our review page?"

Complaint Response Path: "A customer reported a problem. Start with a genuine apology. Ask what we can do to help. Offer a specific next step you can take immediately. Sound like you care about solving this."

Sample output: "I'm truly sorry you're experiencing this. I want to help you get your package. Can you reply with your order number? I'll personally check your tracking right now and follow up with an update in 2 hours."

Suggestion Response Path: "A customer shared an idea. Thank them for thinking about your business. Confirm you've documented it. Tell them how suggestions are reviewed (e.g., 'Our product team reviews all suggestions monthly'). Sound grateful and professional."

Sample output: "Thank you for this suggestion! We love hearing ideas from our customers. I've documented this and passed it to our product team. We review suggestions monthly and will let you know if we move forward."

Same workflow. Three different starting prompts. Three different outputs. All appropriate to the input type.

Ask your AI Buddy: "Help me define the input types for my workflow. What are the main types of input my system receives? For each type, what's the key difference in how I should respond? Let's list minimum three types and then write a tight classifier system prompt that labels each one with a single word."
Key Insight

The classifier is the foundation. If it can't label correctly, nothing else works. A router with bad input produces bad routing. Responses built on bad classification create confusion. Every decision system's success depends on getting the classification right first. That's a leadership lesson too. You can't route to the right solution if you haven't diagnosed the problem correctly. Diagnosis first. Then routing. Then response.

Brief 3

The Tiny Decision Canvas

Design on Paper First

Here's the hard truth: if you can't sketch your logic on paper, you can't wire it in a system.

Before you open your automation platform, grab a piece of paper. Actually grab it. Or open a doc. Something you can write on.

The Tiny Decision Canvas is simple. For each input type you identified, write two things:

First: What does this type mean?

A clear definition. Not vague. Specific.

Example: "PRAISE: The customer expresses satisfaction, thanks, or positive emotion. Not reporting a problem. They're happy."

Not: "PRAISE: Good feedback."

Second: What happens next?

The action. The response. The follow-up.

Example: "Action: Thank them warmly. Ask if they'd consider a public review. Log as positive sentiment. Follow up: Add to customer success file for renewal conversations."

Not: "Send a response."

Why? Because if your logic is too complex to draw on a page, it's too complex to automate. You'll get confused. Your classifier will get confused. Your router will send things to the wrong place. Your responses will miss the point.

A one-page canvas forces clarity. It forces you to answer hard questions before you build.

Common Mistakes

Mistake 1: Too many categories. You defined eight types of input. Eight is too many. More than five is a problem. Your classifier will get confused. Your router will have too many paths. Simplify. Combine types that are similar. Narrow to the essential differences.

Mistake 2: Categories that overlap. You defined COMPLAINT and ISSUE. A customer reports a technical issue. Is that COMPLAINT or ISSUE? Both? Neither? If something could fit two categories, you haven't defined them clearly. Go back. Clarify the boundary between categories.

Mistake 3: Missing the action step. You identified types but didn't think through what to do. "We have PRAISE, COMPLAINT, and SUGGESTION." Great. For each one, what happens next? If you can't answer that, you're not ready to build.

Fix these before you move forward.

Worked Example: Jordan's Tiny Decision Canvas

INPUT TYPE: PRAISE

Definition: The customer expresses satisfaction, thanks, or positive emotion about their experience or service. They might mention what specifically went well. This is not a suggestion for improvement. It's appreciation.

Action: Thank them warmly. Ask if they'd consider sharing this as a public review. Log as positive sentiment. Follow-up: Add to customer success file for renewal conversations.

INPUT TYPE: COMPLAINT

Definition: The customer reports a problem, issue, or dissatisfaction. Something your company did or failed to do caused harm. They're frustrated. They want it fixed.

Action: Apologize. Understand the problem. Offer immediate next step. Take ownership. Follow-up: Track resolution. Confirm closed with customer.

INPUT TYPE: SUGGESTION

Definition: The customer shares an idea for improvement, new feature, or different way of doing business. They're thinking about your future. Not complaining about the present.

Action: Thank them. Document the idea. Tell them how ideas are reviewed. Follow-up: Route to product team. Close loop with customer on decision.

Key Insight

The canvas is your blueprint before build. It forces you to think through logic before complexity hits. A confused canvas becomes a broken workflow. Clarity on paper becomes clarity in code. Design first. Build second.

Brief 4

Build the Logic Layer

Step-by-Step Walkthrough

You have your canvas. You understand the pattern. Now you wire it in. This is the technical build. You're taking your Session 3 workflow and adding intelligence.

Your original workflow looked like this:

Step What It Does
1. Trigger New feedback arrives (Slack message, form submission, email)
2. Data Capture the full text of the feedback
3. AI Response Claude writes a response using a single system prompt
4. Action Send the response (reply to Slack, email response, etc.)
5. Log Record the interaction (timestamp, feedback text, response sent)

Your new workflow adds logic:

Step What It Does
1. Trigger New feedback arrives
2. Data Capture the full text of the feedback
3. Classifier AI Claude classifies input as PRAISE, COMPLAINT, or SUGGESTION
4. Router If/then logic directs to appropriate response path
5a. Praise Response Claude uses praise-specific prompt. Outputs warm thank you.
5b. Complaint Response Claude uses complaint-specific prompt. Outputs solution-focused reply.
5c. Suggestion Response Claude uses suggestion-specific prompt. Outputs gratitude and documentation confirmation.
6. Action Send the appropriate response (reply to Slack, email response, etc.)
7. Log Record interaction with type, classification, and response text

In Your Automation Platform

This is what it looks like in Make, Zapier, or equivalent:

Module 1: Trigger - Your webhook or form trigger. Same as before.

Module 2: Parse Data - Extract the feedback text. Same as before.

Module 3: Classifier Step - Call Claude with tight classifier prompt. Output: one word (PRAISE, COMPLAINT, or SUGGESTION).

Module 4: Router (Conditional Logic) - If classifier output contains "PRAISE", continue to Module 5a. If "COMPLAINT", continue to Module 5b. If "SUGGESTION", continue to Module 5c.

Module 5a, 5b, 5c: Response Paths - Each path calls Claude with a different system prompt. Each gets a different response.

Module 6: Action - Send the response back to Slack/email/etc. This step is the same for all three paths.

Module 7: Log - Record the interaction. Log the classification type so you can analyze patterns later.

The beauty: from the customer's perspective, it looks like one workflow. From the system's perspective, it's three parallel paths. Efficiency plus intelligence.

Pro Tip

Test each path separately before you wire them together. Test the classifier first. Verify it outputs clean one-word labels. Test each response path independently. Only after all three work do you add the router logic.

Key Insight

The router doesn't think. It just directs. All the intelligence is in the classifier. The router is dumb logic (if A then B). Don't expect the router to be smart. Make the classifier smart and the router simple.

Brief 5

Test and Validate

Required for Certification

Your Final Project

Build your complete Classify-Route-Respond workflow. Test it. Document your results. Submit your work.

This isn't theoretical. This is your actual workflow, upgraded, tested, and ready for your team to use.

Part 1: Tiny Decision Canvas

Design

Create your decision canvas on one page. For each input type you identified (minimum three, maximum five), write:

  • A clear definition that distinguishes this type from others
  • A specific action or set of actions that should happen for this type

A colleague should be able to read this and understand your logic flow immediately.

Part 2: Classifier Prompt and Testing

Build

Write your classifier system prompt. It should output ONLY one clean word per input. No explanation.

Test your classifier with minimum five different inputs (including one edge case). Document the input and the output for each.

All outputs should be clean labels. One word. No explanation. No reasoning.

Part 3: Workflow Map

Design

Describe or draw your complete workflow end-to-end. Include all seven steps:

  • 1. Trigger: What starts your workflow?
  • 2. Data: What gets captured?
  • 3. Classifier: What classifies the input?
  • 4. Router: Your three conditional paths
  • 5a/5b/5c. Responses: What happens on each path?
  • 6. Action: How does output go out?
  • 7. Log: What gets logged?

A colleague should be able to read your workflow map and understand how logic flows.

Part 4: Test Results Documentation

Validate

Run three complete workflow tests. One input per category.

For each test, document:

  • Input: What did the customer say?
  • Classified As: What did the classifier output?
  • Path Taken: Which path did the router direct to?
  • Final Response: What was the output?
  • Assessment: Was this appropriate for the input type?

Part 5: Edge Case Testing

Quality

Identify one edge case. An input that's ambiguous or tricky. Run it through your system.

Document:

  • Edge Case Input: What made this tricky?
  • Classified As: What did the classifier output?
  • Was it correct?: Did it classify appropriately?
  • What you learned: Did this reveal any gaps?

This shows you tested beyond happy path.

Grading Rubric

Criteria Excellent (4) Good (3) Needs Work (2) Incomplete (1)
Tiny Decision Canvas Three to five types. Clear definitions. Specific actions. One page. No overlap. Types defined with some specificity. Mostly clear actions. Types listed but definitions vague. Actions unclear. Canvas missing or incomplete.
Classifier Prompt Tight prompt outputs clean labels. Tested with 5 examples. All outputs are one word. Good prompt. Mostly clean labels. Testing complete. Prompt allows some explanation. Testing incomplete. Prompt missing or unclear.
Workflow Map Clear end-to-end description. All seven steps included. Easy to follow. Most steps described. Logic mostly clear. Steps described but some confusion on routing. Map missing or incomplete.
Test Results Three clear tests (one per category) with all elements. Input, classification, path, response, assessment. Tests documented. Most elements present. Testing partial or unclear. No meaningful testing.
Edge Case Testing One clear edge case identified, tested, classified, and assessed. Edge case identified and tested. Assessment clear. Edge case mentioned but not fully tested. No edge case testing.

Before You Submit

  • I've completed my Tiny Decision Canvas with minimum three input types
  • Each type has a clear definition that distinguishes it from others
  • Each type has a specific action that's concrete, not vague
  • I've written a classifier system prompt that outputs one word only
  • I've tested my classifier with five different inputs including one edge case
  • My classifier outputs clean labels (one word, no explanation) consistently
  • I've set up the router logic in my automation platform
  • I've tested that inputs route to the correct paths
  • I've written three distinct response prompts (one for each path)
  • I've tested all three paths with realistic examples
  • Each response is appropriate in tone and action for its input type
  • I've set up logging to capture type, response, and timestamp
  • I can explain my workflow map clearly to a colleague
  • My submission includes all five parts
Wrap-Up

Key Takeaways

  • Before AI responds, it needs to know what it's responding to. Classification comes before response. Every time. This is a logic principle and a leadership principle. Listen first, then respond.
  • The classifier is the foundation. If it can't label correctly, nothing else works. Bad classification breaks routing. Bad routing creates bad responses. Get classification right first.
  • The router doesn't think. It just directs. All the intelligence is in the classifier. The router is dumb logic (if A then B). Don't expect the router to be smart. Make the classifier smart.
  • Design on paper before you wire in a system. If you can't sketch your logic on one page, the system is too complex. Simplify before you build.
  • This pattern works across all organizations. Classify. Route. Respond. Whether you're in operations, delivery, or leadership, this pattern applies. It's not mission-specific. It's fundamental.
  • Personalization at scale. You're now treating different inputs differently automatically. Different tones. Different actions. Different responses. Same efficiency. That's maturity in automation.
  • You're adding intelligence, not just speed. An automated system without classification is fast but dumb. Fast and dumb is the opposite of useful. Add the logic layer and you've got something that works.
  • Test edge cases, not just happy paths. Your system works when everything is clear and fits your categories. Real life is messy. Test what breaks. Test ambiguous inputs. Fix those before your team uses this.
