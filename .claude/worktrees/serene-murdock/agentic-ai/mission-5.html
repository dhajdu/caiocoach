Learning Guide

Bring your Mission 4 workflow. You have a logic layer that works for predictable cases. Somewhere in that logic, there's an edge case that makes the rules more complicated. That's where your agent lives.

Work from your real system. The agent you design here is for your actual workflow. Not a hypothetical. Not a practice example. The problem you're solving is real.

Design first, build second. Before you code anything, design your agent. Mission statement. Instructions. Guardrails. Escalation criteria. The design is 80% of the work.

Guardrails are non-negotiable. An agent without guardrails is a liability. Define what it can access, what it can promise, when it escalates. Be specific.

Rules of the Road

  • Start with a clear agent mission. One sentence. What is it trying to achieve?
  • Identify brain, tools, and memory before you build. What does your agent need to know and do?
  • Test your agent three ways: happy path, edge case, adversarial. Does it hold its guardrails?
  • Define success metrics before you deploy. What's the baseline? What's your target?
  • Design human-in-the-loop from the start. Agent handles routine. Human handles judgment.
Pro Tip

Use one Claude Project for all five briefs. Start with your Mission 4 workflow context. Ask Claude: "I want to build an agent for [your situation]. Where does my decision logic break?" Let Claude help you identify the agent opportunity before you start designing.

Brief 1 Teaching

The Ceiling of Logic

Decision trees are powerful tools for handling predictable inputs. You define the rules. You design the branches. "If input is X, do Y." This works great when you can predict every input variation. But the moment inputs become too varied, too complex, or too unpredictable for your decision tree, you hit the ceiling of logic.

The shift from logic to agents is not a question of "which is better." It's a question of "which tool is right for this problem."

When Decision Logic Excels

  • Well-defined inputs with predictable variations
  • Clear categorization rules
  • Limited edge cases
  • Cost-sensitive operations
  • Fully predictable workflows

When Decision Logic Breaks

  • Inputs are too varied or unpredictable
  • Nuance matters more than categorization
  • Edge cases outnumber standard cases
  • Human judgment is required
  • The decision tree becomes harder to maintain than doing the work manually

The Shift

Instead of telling the system what to do in every situation (decision logic), you tell it what to achieve (agent goal). The system figures out how to get there.

Worked Example: Customer Support

Imagine you have a customer support workflow. A customer submits feedback. Your logic layer classifies it based on keywords:

  • If it contains "broken" and "product," route to Product Team
  • If it contains "billing," route to Finance Team
  • If it's grateful language, send thank you

The Happy Path: This works for 80% of inputs.

The Edge Cases: What about the customer who says "I love your product but can't afford the pricing"? That's both a compliment and a concern. The categorization breaks. What about the customer who reports a bug that also involves a billing dispute? Multiple teams. What about the customer whose frustration is implied but not explicit? The rule fails.

Without an Agent

You add more branches. You complicate the tree. Each new edge case is another rule to manage. Maintenance becomes unbearable.

With an Agent

You define the goal: "Understand this customer's core concern and route them to the right team." The agent figures out that this is a pricing concern, not a bug. Reads between the lines. Prioritizes by urgency. Handles ambiguity.

Key Insight

Decision logic is for problems you can fully predict. Agents are for the ones you can't. Rules follow paths. Agents find solutions.

Starter Prompt for Your Work

"Look at your Mission 4 workflow and the logic layer you built. Which inputs or situations made that logic complicated? Which required you to add extra branches? Which edge cases did you have to design around? Those are your agent opportunities."

Brief 2 Teaching

The Agent Loop: Think-Plan-Act-Reflect

The agent loop is the cycle that allows agents to handle complexity. It's not a simple input-to-output. It's iterative. It allows for course-correction. It allows the agent to reflect on whether it achieved the goal.

T
Think

The agent reads the input and understands what's being asked. What is the user trying to accomplish? What are the constraints? What does success look like? The agent builds a mental model of the problem.

P
Plan

Based on that understanding, the agent decides what steps to take. What data does it need? What tools should it use? In what order? What's the strategy? The agent creates a plan to achieve the goal.

A
Act

The agent executes the plan. It uses the tools. It gathers the information. It takes action. It moves toward the goal.

R
Reflect

The agent evaluates the output. Did it work? Is the goal achieved? Is the output good enough? If yes, it's done. If not, it loops back. It adjusts the plan. It tries a different approach.

This loop is why agents can handle complexity that would break a decision tree. A decision tree follows a single path. An agent finds a solution. It can course-correct. It can try again. It can handle the unexpected because it's not rigidly following a branch - it's pursuing a goal.

Worked Example: Customer Support Agent Using the Loop

Customer submits: "We've been using your product for 2 years. Last week the reports feature stopped working. Our reports are mission-critical. We need this fixed today."

Think: The agent understands: (1) Long-term, valuable customer. (2) Critical feature broken. (3) Time-sensitive. (4) Likely technical issue requiring investigation. (5) High escalation priority.

Plan: The agent decides: (1) Retrieve customer account history to understand their usage. (2) Search the knowledge base for known issues with reports. (3) Check if there's a recent system update that might have caused this. (4) If no quick fix, prepare escalation to engineering with all context.

Act: The agent queries the customer database. Finds they're a VIP account. Searches the knowledge base. Finds a post about a recent update and a workaround. Tries the workaround. Realizes it won't work for their use case. Prepares escalation documentation.

Reflect: The agent checks its output. Is the goal (resolve or properly escalate) achieved? Yes. The customer has a workaround path and a clear escalation if that doesn't work. If the reflection revealed the goal wasn't achieved, the loop would restart.

Key Insight

A decision tree follows a path. An agent finds a solution. The loop enables the search.

Brief 3 Teaching

Agent Anatomy: Brain + Tools + Memory

Every agent has three components. All three must be strong for the agent to work.

BRAIN
The LLM

The underlying intelligence. Claude. GPT. The language model that does the thinking, planning, and reflecting. The better the brain, the better the reasoning. The more powerful the brain, the more complex the problems it can handle.

TOOLS
Capabilities

What the agent can do beyond thinking. Web search, API connections, file access, calculations, database queries, email sending. An agent without tools is just a chatbot. It can think, but it can't do anything. Tools are what let it act.

MEMORY
Context & Knowledge

The data the agent draws from. Reference documents, conversation history, customer data, rules and guardrails. The more complete and accurate the memory, the better the agent's output.

ALL THREE
Working Together

A great brain with weak tools can think but not act. Great tools with poor memory make mistakes. Poor memory combined with weak reasoning leads to confidently wrong answers. All three must be strong.

Worked Example: Anatomy Check

Agent Mission: "Understand customer technical issues, gather information, diagnose common problems, and escalate complex cases to the right specialist with full context."

Brain: Claude 3.5 Sonnet. Good reasoning for diagnosis and prioritization.

Tools: (1) Search knowledge base for known solutions. (2) Retrieve customer account history. (3) Check system status and recent updates. (4) Send escalations to engineering.

Memory: (1) Knowledge base with known issues and solutions. (2) Customer database with account types and history. (3) System documentation and recent changes. (4) Escalation protocols and specialist contact info.

Key Insight

Your agent is only as good as its weakest component. A brilliant brain with no access to data is a wasted brain. Access to tools without guardrails is dangerous. Memory without context is noise.

Build Your First Agent

Step-by-Step Agent Build

1

Define Your Mission (5 min)

Write one sentence. What is your agent's goal? Who is the user? What does success look like? Template: "This agent helps [user] achieve [specific goal] by [main method]."

5 min
2

Write the Instructions (5 min)

How should it behave? What tone? What should it always do? What should it never do? These are your guardrails embedded in instructions.

5 min
3

Identify Your Knowledge Base (5 min)

What data does your agent need? Process documents? FAQ? Customer data? Product information? Your agent is only as good as what it knows.

5 min
4

Add Your Tools (5 min)

What does your agent need to do? Search knowledge base? Retrieve data? Calculate? Connect to systems? Be specific about what each tool can access.

5 min
5

Write Your Guardrails (5 min)

What is off-limits? Data access, decisions, communications, tone. Write specific guardrails for each category. These are non-negotiable.

5 min

Worked Example: Complete Build

Selected Item: Lead Qualification for Sales

Mission: "This agent qualifies inbound leads for the sales team by analyzing their company profile, engagement level, and fit with our target customer profile, then routing them to the right sales representative."

Instructions:

  • Read every lead inquiry carefully. Understand what the prospect needs.
  • Check our target customer profile in the knowledge base.
  • Analyze: company size, industry, growth stage, problem fit, engagement level.
  • Use a professional and helpful tone. These are potential customers.
  • Ask clarifying questions if the fit isn't clear.
  • Document your analysis as you go.

Tools:

  • Search knowledge base for customer profile and targeting rules
  • Retrieve prospect company data from LinkedIn API
  • Calculate fit score based on target customer profile
  • Access sales CRM to find the right sales rep

Guardrails:

  • Never tell a prospect our pricing without checking with sales
  • Never promise timelines or features
  • Can access company information but not personal data
  • Never share our customer list or target customer profile with prospects

Escalation Criteria:

  • If the prospect asks about pricing, escalate to sales
  • If the prospect is a competitor, escalate to management
  • If fit score is below 30%, ask for human review before routing
  • If the inquiry is about a partnership or integration, escalate to partnerships
Key Insight

You are not doing the work. You are designing the system. Supervisor, not spectator.

Brief 5 Validation

Guardrails and Human in the Loop

How to Test an Agent

Testing an agent is different from testing a workflow. You're not checking if the agent follows a path. You're checking if it achieves the goal within the guardrails.

Happy Path Testing: Give it a straightforward request that fits your agent's mission perfectly. Does it do what you designed it to do? Does it produce good output?

Edge Case Testing: Give it a request that's not in your data, or partially in your data. A request that's ambiguous or contradictory. Does the agent handle it gracefully? Does it ask for clarification? Does it escalate appropriately?

Adversarial Testing: Try to break it. Ask it to violate its guardrails. Give it conflicting instructions. Attempt to trick it into sharing information it shouldn't. Does it hold its guardrails? Does it escalate when it should?

Pro Tip

Guardrails are non-negotiable. Test them relentlessly. If your agent can be tricked into violating a guardrail, adjust the guardrail language or add additional safety checks. A guardrail that fails under adversarial testing is not actually a guardrail.

Success Metrics (Determine Phase)

Before you deploy, you need to know what success looks like. This is the Determine phase of the 5D Framework. You're determining your success metrics before you go live.

Success metrics are not vanity metrics. They're the weekly numbers you'll review to know if the agent is working.

Example bad metrics: "Agent is live." "Lots of people are using it."

Example good metrics:

  • Support escalation time reduced from 2 hours to 30 minutes
  • Correct diagnosis on first attempt: 75% of cases
  • Customer satisfaction for agent-resolved issues: 4.2/5 or higher
  • Human escalations reduced by 40%
  • Agent-to-human ratio improved from 30/70 to 60/40

The Determine Phase Questions:

  1. What is the current baseline for this process (before the agent)?
  2. What improvement are you targeting? (Faster? More accurate? Better experience?)
  3. How will you measure that improvement?
  4. What are your weekly metrics?
  5. What's your acceptable variance? (Is 70% success enough, or do you need 85%?)
  6. When do you escalate the agent for retraining? (If success drops below X%, you adjust guardrails or knowledge base.)

Human-in-the-Loop Design Patterns

Design your human-in-the-loop workflow before you deploy. Choose the pattern that fits your risk tolerance and scalability needs.

PATTERN 1
Agent Drafts, Human Approves

The agent produces output (email, analysis, recommendation). A human reviews and approves before it goes live. Good for high-stakes decisions or customer-facing communications.

PATTERN 2
Agent Routine, Human Edge Cases

The agent manages standard cases (80% of requests). When the agent detects an edge case or is unsure, it escalates to a human. Good for scalability.

PATTERN 3
Agent Research, Human Decides

The agent gathers information and presents options. The human makes the decision. Good for complex decisions requiring human judgment.

PATTERN 4
Agent Monitors, Human Acts

The agent monitors for issues or opportunities. When it finds one, it alerts a human, who takes action. Good for continuous improvement.

Key Insight

The agent doesn't work alone. Agent handles routine, human handles judgment. Agent drafts, human decides. This is the 50/50 era. Not agent-only automation. Agent plus human working together.

Key Takeaways

  • Decision logic solves predictable problems. Agents solve variable ones. Decision trees work when you can predict the inputs. Agents work when you can't. Know which problem you're solving.
  • The Think-Plan-Act-Reflect loop is why agents work. They don't follow a single path. They pursue a goal. They course-correct. They try again. That's how they handle complexity.
  • Agent anatomy: Brain (LLM) + Tools (capabilities) + Memory (data). Every agent needs all three. A great brain with no tools is a chatbot. Tools without guardrails are dangerous. Memory shapes every decision.
  • Guardrails are how you stay in control. An agent without guardrails will make decisions you never authorized. Define what it can access, what it can promise, when it escalates. Non-negotiable.
  • Escalation is not failure. It's by design. The agent handles routine. Humans handle judgment. Agent drafts, human decides. This is the 50/50 era. Not agent-only. Not human-only. Both.
  • Your role is designer, not executor. You design the mission, guardrails, and escalation rules. The agent executes. You supervise the system, not the work. That's the leadership shift.
  • Success metrics matter more than the agent itself. Before you deploy, know what success looks like. What's the baseline? What's your target? How will you measure it weekly? That's your Determine phase.
  • You've built the full stack. From problem statement to packaged AI to workflow to logic to agent. Mission 6 is the final step: the production brief. That's what takes this from your laptop to your organization.
